# 2.4 Реализация алгоритма построения области эффективных решений

Разработанное приложение реализует интеллектуальный механизм многокритериального анализа, позволяющий формировать несколько допустимых вариантов распределения задач по командам.

Необходимость использования подобного подхода обусловлена высокой сложностью современных IT-проектов, в рамках которых требуется одновременно учитывать ограничения по специализации исполнителей, стоимость выполнения работ, уровень загрузки команд и приоритетность задач для бизнеса. В подобных условиях поиск единственного «идеального» решения становится недостаточно эффективным, поскольку улучшение одного показателя зачастую приводит к ухудшению другого.

В теоретической основе проекта используется **концепция Парето-эффективности**: решение считается эффективным, если невозможно улучшить один критерий без ухудшения хотя бы одного другого. На практике система формирует **набор из четырёх альтернативных сценариев** распределения с различными весами критериев (точки **A, C, D, F**), что позволяет руководителю проекта сравнить компромиссные варианты и выбрать наиболее подходящий с учётом текущих условий.

---

## Программная реализация

Для реализации механизма интеллектуального распределения задач в системе разработан аналитический модуль оптимизации:

| Компонент | Файл | Назначение |
|-----------|------|------------|
| Алгоритм распределения | `src/utils/optimizer.js` | Расчёт вариантов и метрик |
| API оптимизации | `src/controllers/taskController.js` | Запуск и выдача результатов клиенту |
| Нормализация приоритета | `src/utils/businessPriority.js` | Приведение `business_priority` к шкале 1–3 |
| Визуализация | `src/controllers/visualizationController.js` | Графики и сравнение сценариев |
| Персональные рекомендации | `src/controllers/preferenceController.js` | Отдельный расчёт по весам пользователя |

> Класс `Optimizer` реализует **жадный эвристический** алгоритм назначения с взвешенной целевой функцией.

---

## Входные данные алгоритма

На вход подаются:

- **список задач проекта** (наименование, описание, тег направления, сложность, срок, статус);
- **перечень команд** (тег, стоимость единицы работы, ёмкость, текущая загрузка);
- **бизнес-приоритет задачи** `business_priority` ∈ {1, 2, 3} (низкий / средний / высокий);
- **показатель сложности** `complexity` (шкала 1–10).

Дополнительно в интерфейсе хранятся **пользовательские настройки** (`UserPreference`: веса α, β, γ, предпочитаемые команды), которые используются в модуле персональных рекомендаций и **не подмешиваются** в основную целевую функцию кнопки «Оптимизировать».

---

## Этап 1. Проверка допустимости назначений

Перед расчётом критериев система определяет допустимые пары «задача — команда»:

1. **Совместимость по специализации:** тег задачи (`frontend`, `backend`, `ML`) должен совпадать с тегом команды.
2. **Ограничение по ёмкости:** при назначении задачи с трудоёмкостью `complexity` суммарная загрузка команды не должна превышать `capacity`:

\[
\sum_{j \in J_i} \text{complexity}_j \leq K_i
\]

где \(J_i\) — множество задач, назначенных команде \(i\), \(K_i\) — ёмкость команды.

Несовместимым парам в целевой функции присваивается большой штраф, исключающий их из выбора.

---

## Этап 2. Многокритериальный анализ

Для каждой допустимой пары *(задача j, команда i)* оцениваются **три критерия**.

### Критерий 1. Стоимость (вес α)

\[
C_{ij} = \text{complexity}_j \cdot \text{cost}_i
\]

### Критерий 2. Загрузка (вес β)

Для пары «задача — команда»:

\[
L_{ij} = \frac{\text{complexity}_j}{K_i}
\]

После распределения для оценки сценария используется **максимальная относительная загрузка** среди команд:

\[
L_i = \frac{T_i}{K_i}, \quad T_i = \sum_{j \in J_i} \text{complexity}_j
\]

\[
L_{\max} = \max_i L_i
\]

Если \(L_{\max}\) превышает допустимый уровень, сценарий считается перегруженным; при жадном назначении новая задача не назначается команде, у которой \(T_i + \text{complexity}_j > K_i\).

### Критерий 3. Предпочтительность / бизнес-приоритет (вес γ)

Третий критерий отражает **только приоритет задачи для бизнеса** \(P_j \in \{1, 2, 3\}\), заданный в карточке задачи. Стоимость команды, профиль менеджера и прочие эвристики в этот критерий **не входят**.

\[
\text{preference}(P_j) = P_j
\]

---

## Этап 3. Нормализация критериев

Стоимость и загрузка измеряются в разных единицах, поэтому для них применяется **min–max нормализация** по всем допустимым парам текущего расчёта:

\[
\text{costNorm}_{ij} = \frac{C_{ij} - C_{\min}}{C_{\max} - C_{\min}}, \qquad
\text{loadNorm}_{ij} = \frac{L_{ij} - L_{\min}}{L_{\max} - L_{\min}}
\]

Для бизнес-приоритета используется **фиксированная** нормализация на шкале 1–3 (не зависит от состава команд):

\[
\text{prefNorm}(P_j) = \frac{P_j - 1}{2} \in [0,\, 1]
\]

| \(P_j\) | Смысл | \(\text{prefNorm}\) |
|--------|--------|-------------------|
| 1 | Низкий приоритет | 0 |
| 2 | Средний | 0,5 |
| 3 | Высокий | 1 |

Для одной задачи значение \(\text{prefNorm}\) одинаково у всех совместимых команд: приоритет характеризует важность **задачи**, а не предпочтение конкретной команды.

---

## Этап 4. Агрегированная целевая функция

Общий критерий для пары *(j, i)* минимизируется (чем меньше значение, тем предпочтительнее назначение при данных весах):

\[
F_{ij} = \alpha \cdot \text{costNorm}_{ij} + \beta \cdot \text{loadNorm}_{ij} - \gamma \cdot \text{prefNorm}(P_j)
\]

где:

- \(\text{costNorm}_{ij}\), \(\text{loadNorm}_{ij}\), \(\text{prefNorm}(P_j)\) — нормализованные показатели;
- \(\alpha\), \(\beta\), \(\gamma\) — неотрицательные весовые коэффициенты, \(\alpha + \beta + \gamma = 1\);
- знак «минус» перед \(\gamma \cdot \text{prefNorm}\) обеспечивает **максимизацию** вклада бизнес-приоритета при минимизации \(F_{ij}\).

Изменение весов смещает приоритеты системы: при большом \(\alpha\) доминирует экономия, при большом \(\beta\) — выравнивание загрузки, при большом \(\gamma\) — назначение в пользу задач с высоким \(P_j\).

---

## Этап 5. Формирование набора сценариев (область компромиссов)

Алгоритм последовательно выполняет **четыре прогона** с фиксированными весами, имитируя обход компромиссов между критериями:

| Точка | Название сценария | α | β | γ |
|-------|-------------------|---|---|---|
| **A** | Минимизация стоимости | 0,8 | 0,1 | 0,1 |
| **C** | Равный баланс | 0,33 | 0,33 | 0,34 |
| **D** | Акцент на разгрузке | 0,3 | 0,6 | 0,1 |
| **F** | Максимум приоритета | 0,1 | 0,2 | 0,7 |

Внутри каждого прогона задачи группируются по тегу и обрабатываются в порядке убывания:

\[
\text{sortKey}_j = P_j \cdot \text{complexity}_j
\]

Для каждой задачи выбирается команда с **минимальным** \(F_{ij}\) среди допустимых и не перегруженных исполнителей (жадная стратегия).

Итоговые метрики сценария:

| Метрика | Описание |
|---------|----------|
| `totalCost` | \(\sum C_{ij}\) по всем назначениям |
| `maxLoad` | \(L_{\max} \cdot 100\,\%\) |
| `totalPreference` | \(\sum_j P_j\) — сумма бизнес-приоритетов назначенных задач |

---

## Этап 6. Визуализация и выбор решения

Результаты передаются на клиент и отображаются в модуле оптимизации: карточки сценариев A / C / D / F, сводные показатели стоимости, загрузки и суммы приоритетов, таблица назначений. Пользователь сравнивает варианты и применяет выбранный сценарий к проекту.

При изменении состава задач, команд, приоритетов или сложности достаточно повторно запустить оптимизацию — система пересчитает все четыре сценария на актуальных данных.

---

## Персональные рекомендации (отдельный контур)

Настройки менеджера (`weightCost`, `weightLoad`, `weightPreference`, `preferredTeamIds`) применяются в эндпоинте персональных рекомендаций (`GET /api/visualization/recommendations`) и **не заменяют** описанную выше целевую функцию основной оптимизации. Это позволяет разделить:

- **объективный многокритериальный расчёт** по данным задач и команд;
- **субъективный профиль пользователя** для дополнительной подсказки.

---

## Вывод

Реализованный механизм представляет собой аналитический модуль многокритериального распределения задач по командам с учётом стоимости, загрузки и бизнес-приоритета (1–3). Формирование четырёх взвешенных сценариев даёт руководителю наглядную область компромиссных решений и поддерживает принятие управленческих решений в условиях меняющихся параметров IT-проекта.

- стоимость команды в формуле предпочтительности;
- взвешенная сумма «предпочтительности менеджмента и команд» (\(w_1\), \(w_2\));
- поля `preferredTags`, `minPreferenceThreshold` в основном прогоне оптимизации.

## Пример расчёта

Задача: `complexity = 8`, `business_priority = 3` → \(\text{prefNorm} = 1\).  
При \(\gamma = 0{,}7\) вклад приоритета в \(F_{ij}\): \(-0{,}7 \cdot 1\), что снижает агрегированный критерий и повышает привлекательность назначения такой задачи при прочих равных.
